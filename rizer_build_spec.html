RIZER Procurement & Inventory — New Workspace Build
Complete ground-up rebuild. Follow build order exactly. Do not build on live account.
Build order
1
Create new CarbonWeb dev workspace
All build and testing happens here. Live account stays untouched until final migration.
2
Build Inventory board
Foundation board — all other boards reference it. Build columns and structure first.
3
Build Procurement Requests board + WorkForm
Parent board with subitem structure. Build Path A and Path B form logic.
4
Build Maintenance Orders board
Add columns, groups, single-direction automations only. No status↔group loops.
5
Build Construction Orders board
Same structure as Maintenance. Add property/building column.
6
Build Direct Ship board
Simpler board — no inventory deduction. Connect to Procurement.
7
Build Maintenance Order Report board
Report board populated by single automation trigger only.
8
Build Construction Order Report board
Same pattern as Maintenance Report.
9
Build Enterprise Workflows (1174, 1192, 1234 replacements)
Build native workflows after all boards exist. They reference cross-board connections.
10
Build Make.com scenario 1230 with idempotency check
Inventory math scenario — only scenario staying in Make.com. Add data store guard first.
11
Build Returns workflow
Permission-gated. Reverses inventory deduction, flags accounting, logs audit trail.
12
Generate office-prefilled form URLs
One form, UTM-prefilled per Nebraska office. Document in mondayDoc.
13
End-to-end test in dev
Submit form → triage → inventory deduction → report. Verify single deduction, single report row, single notification.
14
Migrate to live
Only after full dev validation. Reconcile inventory counts with Audrey before go-live.

Automations — status transitions (single direction only)

Never add a "When item moved to group → set Status" automation. This creates the loop. Status drives group movement. Group movement never drives status.
TRIGGER: When Status changes to Order Received ACTION: Move item to Order Received group TRIGGER: When Status changes to Ready to Ship ACTION: Move item to Ready to Ship group TRIGGER: When Status changes to Shipped ACTION: Move item to Shipped group

Automations — date setters

TRIGGER: When Received Amount changes ACTION: Set Receiving Date = today TRIGGER: When Status changes to Shipped ACTION: Set Shipping Date = today

Automations — report creation (single trigger only)

TRIGGER: When Status changes to Shipped ACTION: Create item in Maintenance Order Report MAP: Item name, Order Date, Shipping Date, Receiving Date, Ordered Amount, Received Amount, Item Price, Main Property, Building and Unit Number, Request ID DO NOT ADD: When Receiving Date changes → create report item DO NOT ADD: When Shipping Date changes → create report item

Automations — delivery timeline

TRIGGER: When item created AND Delivery Timeline = ASAP ACTION: Set Deadline = today + 1 day, Set Order Date = today TRIGGER: When item created AND Delivery Timeline = 24-48 hours ACTION: Set Deadline = today + 2 days, Set Order Date = today TRIGGER: When item created AND Delivery Timeline = 1 week ACTION: Set Deadline = today + 7 days, Set Order Date = today TRIGGER: When item created AND Delivery Timeline = 2 weeks ACTION: Set Deadline = today + 14 days, Set Order Date = today ONE automation per delivery option only. Do not duplicate.

Automations — status mirror from Procurement

TRIGGER: When Maintenance Order Status (mirror) changes to Order Received ACTION: Set Status = Order Received TRIGGER: When Maintenance Order Status (mirror) changes to Ready to Ship ACTION: Set Status = Ready to Ship TRIGGER: When Maintenance Order Status (mirror) changes to Shipped ACTION: Set Status = Shipped

Webhooks (Make.com triggers)

WEBHOOK 1: When item is created → send webhook (triggers scenario 1230) WEBHOOK 2: When Status changes to Shipped → send webhook (triggers scenario 1230) ONE webhook per event. No duplicate pairs.
